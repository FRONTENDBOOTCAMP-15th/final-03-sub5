# 요구사항 정의서

## 1. 문서 정보

- 문서 목적: 러닝 기록 관리 웹앱의 요구사항을 명확히 정의하여 이해관계자 간 공통 이해를 확보한다.
- 대상: 기획, 디자인, 프론트엔드, 백엔드
- 범위: 웹 기반 러닝 기록 및 통계 관리 시스템

## 2. 시스템 개요

사용자가 러닝 운동 기록을 추가·조회·수정·삭제하고, 주간/월간 통계 및 페이스 분석을 통해 운동 성과를 관리하는 웹 애플리케이션이다. 앵커 스크롤 방식의 단일 페이지에서 모든 정보를 확인할 수 있으며, 최근 5개의 러닝 기록을 카드 형태로 제공한다.

## 3. 사용자 유형

- 비회원: 회원가능 후 진행 유도
- 회원: 개인의 러닝 기록을 추가, 조회, 수정, 삭제하고 통계를 확인할 수 있는 사용자

## 4. 기능 요구사항

| ----- | -------- | ------------------------------------------------------------------------------------------------------- |
| FR-01 | 1순위    | 사용자는 앵커 스크롤 네비게이션(홈/기록/분석/최근기록)을 통해 페이지 내 섹션으로 이동할 수 있어야 한다. |
| FR-02 | 1순위    | 사용자는 오늘의 러닝 요약(거리, 시간, 페이스)을 조회할 수 있어야 한다.                                  |
| FR-03 | 1순위    | 사용자는 주간 러닝 거리(1주 적산)를 조회할 수 있어야 한다.                                              |
| FR-04 | 1순위    | 사용자는 월간 러닝 거리(1달 적산)를 조회할 수 있어야 한다.                                              |
| FR-05 | 1순위    | 사용자는 최근 기록 최대 5개를 카드 형태로 조회할 수 있어야 한다.                                        |
| FR-06 | 1순위    | 사용자는 1주간의 평균 페이스 통계를 그래프로 확인할 수 있어야 한다.                                     |
| FR-07 | 1순위    | 사용자는 러닝 기록(날짜, 운동시간, 거리)을 추가할 수 있어야 한다.                                       |
| FR-08 | 1순위    | 시스템은 입력된 거리와 시간을 기반으로 평균 페이스를 자동 계산해야 한다.                                |
| FR-09 | 1순위    | 사용자는 운동 유형을 필수로 선택해야 한다.                                                              |
| FR-10 | 2순위    | 사용자는 기존 러닝 기록을 수정할 수 있어야 한다.                                                        |
| FR-11 | 2순위    | 사용자는 러닝 기록을 삭제할 수 있어야 한다.                                                             |
| FR-12 | 2순위    | 사용자는 운동 장소를 지도 API로 선택하거나 직접 입력할 수 있어야 한다.                                  |
| FR-13 | 2순위    | 사용자는 소모 칼로리와 메모를 선택적으로 입력할 수 있어야 한다.                                         |
| FR-14 | 2순위    | 사용자는 기간별(이번 주/월, 지난 주/월, 전체) 기록을 필터링할 수 있어야 한다.                           |
| FR-15 | 2순위    | 사용자는 기록을 정렬(최신순, 거리순, 페이스순)할 수 있어야 한다.                                        |
| FR-16 | 3순위    | 사용자는 주간/월간 목표 거리를 설정하고 진행률을 확인할 수 있어야 한다.                                 |
| FR-17 | 3순위    | 사용자는 기록 데이터를 CSV/JSON 형식으로 내보낼 수 있어야 한다.                                         |
| FR-18 | 3순위    | 사용자는 외부 데이터를 가져와 기록에 추가할 수 있어야 한다.                                             |

## 5. 비기능 요구사항

| ID     | 우선순위 | 요구사항                                                                               |
| ------ | -------- | -------------------------------------------------------------------------------------- |
| NFR-01 | 필수     | 러닝 기록 데이터는 안전하게 DB에 저장되어야 한다.                                      |
| NFR-02 | 필수     | 페이지 로딩 시 데이터 로딩 상태를 사용자에게 표시해야 한다.                            |
| NFR-03 | 필수     | 입력값 검증을 통해 유효하지 않은 데이터(음수 거리, 미래 날짜 등) 입력을 방지해야 한다. |
| NFR-04 | 권장     | 통계 데이터 조회는 평균 1초 이내 응답을 목표로 한다.                                   |
| NFR-05 | 필수     | 모바일 반응형 디자인을 지원해야 한다.                                                  |
| NFR-06 | 권장     | 오프라인 상태에서 입력한 기록은 임시 저장 후 온라인 시 동기화되어야 한다.              |
| NFR-07 | 필수     | 접근성 기준을 충족해야 한다(키보드 네비게이션, 스크린 리더 지원).                      |

## 6. 제약 및 가정

- 브라우저 기반 웹 앱으로 제공한다.
- 최근 기록은 최대 5개까지 표시하며, 새로운 기록 추가 시 가장 오래된 기록이 목록에서 제거된다.
- 거리 입력 범위는 0.1km ~ 200km로 제한한다.
- 운동 시간은 1분 ~ 24시간 범위로 제한한다.
- 페이스 계산 공식: (운동시간(초) ÷ 거리(km)) → "분:초/km" 형식
- 지도 API는 위치 권한이 필요하며, 권한 거부 시 직접 입력만 가능하다.
- 통계는 사용자의 로컬 타임존 기준으로 계산된다.
